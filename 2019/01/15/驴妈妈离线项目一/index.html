一、驴妈妈旅游网用户行为日志大数据离线分析平台

1.开放周期及人数分工

  1. 2016年，10个人用了9个月

  2. 2019年，5个人用2-3个月

  3. 小组成员：

       + 运维工程师
  + 数据采集工程师
+ 数据etl工程师
+ 数据分析师
+ 可视化工程师
+ 算法工程师

2.项目目的

​ 熟悉完整的离线项目流程（用户日志分析项目）

二、大数据项目开发流程

2.1 项目需求分析

​ 结合公司的业务、数据等情况考虑需要从哪些数据中获取哪些指标

2.2 项目调研

​ 根据公司的需求调研目前大数据领域的主流平台架构

  • 离线sql分析：hive

    • 实时交互sql分析

      kylin

      impala

      sparksql

    • 实时流式计算分析

      sparkstreaming

      storm

      flink

2.3 项目平台架构的设计–大数据平台架构师

  • 概要设计 – 架构输入设计
  • 详细设计 – 具体每个使用到的框架的设计

2.4 项目的编码 - 20%-30%时间

  • etl-mapreduce开放
  • 分析 - hive sql 开发
  • 其他接口 - java
  • shell 脚本

2.5 项目的测试

抽取样本数据放入平台测试

  • 单元测试
  • 联调测试
  • 集成测试

2.6 项目上线

  • 试运营
  • 正式上线

2.7 项目运维

  • 优化
  • hive或spark的优化

2.8 项目的升级

  • 新需求的开发
  • 离线升级为离线+实时分析平台

三、大数据分析平台的分层

3.1 数据采集层

采集层框架:

sqoop 关系型数据库的数据导出

flume 日志导入

datax 查资料

3.2 数据清洗/ETL层

3.2.1 ETL层框架：

  • mapreduce
  • hive – 内置函数 udf、udtf、udaf
  • spark
  • kettle – 国外开源的一款有图形化界面的专业的ETL工具

3.2.2 ETL过程：

  1. 数据清洗
  2. 数据格式转化
  3. 字符串截取或拼接
  4. 脏数据过滤
  5. 与业务需求无关的数据过滤
  6. 数据的解析及补全
  7. 结合项目需求分析

3.2.3 ETL的数据来源

系统本地

RDBMS

HDFS – 本项目数据来源

kafka

hbase

3.2.4 ETl后数据存储

  • RDBMS – 针对数量比较少而且是结构化的数据是采用‘
  • HDFS
  • HDFS - hive映射 ，数据必须是结构化的
  • hbase 海量数据存储，当前用户行为日志数据为非结构化数据

3.3 数据分析层

  • 数据分析师
  • hive工程师
  • spark工程师

3.3.1框架：

  • 离线sql分析 ：hive
  • 实时交互sql分析： kylin、impala、sparksql
  • 实时流式计算分析： sparkstreaming、storm、flink

3.3.2 数据来源：

  • ETl的输出

3.3.3 数据存储：

​ 存储到可以方便数据分享及数据报表可视化的数据库中

+ RDBMS
+ hbase
+ redis
+ mysql -- 当前项目使用

3.3.4 数据的可视化

  • javaweb框架：

    Spring + SpringMVC +mybatis

  • 图形展示插件：

    highcharts/echarts

    SSM+highcharts插件 – 当前项目使用

    hue/zeppline –大数据的web框架

    tableau软件实现

3.3.5数据的深入分析’

  • 推荐模型或预测模型的建立及使用 —机器学习

四、大数据平台分类

4.1 离线大数据分析平台

  • 对分析结果的时效性要求不高，反馈时间不做要求
  • 对服务器的cpu及内存性能要求比较低

4.2 实时大数据分析平台

  • 对分析结果的时效性要求高，反馈时间要求在秒级别

  • 实时分析：

    实时交互查询分析：sparksql、impala、kylin、phoniex-hbase

    实时流式计算分析：

    ​ sparkstreaming/storm/flink

4.3 离线+实时大数据分析平台

  • lambda架构

五、 大数据集群及配置

5.1 服务器选择

  1. 租赁云服务器

    • aliyun、腾讯云、华为、百度云

    • 网速：至少千兆宽带，最好万兆

  2. 公司内部机房

  • 物理服务器的选择：华为、浪潮、（合资）戴尔

5.2 大数据集群服务器的配置

  1. 操作系统：centos7.x
  2. cup 8、16、32、64、128核
  3. 内存: 16g、24、32、40、。。。。256g
  4. 磁盘： 4t、6t、8t、。。。24t

5.3 大数据集群规模及大数据框架部署情况

  1. 集群节点规模：
  • 小型集群：15台以下
  • 中型集群：15-50台
  • 大型集群：50台以上

六、大数据平台的数据来源

  1. 应用服务器的访问日志（Apache、tomcat、nginx、resin）（最传统最常见）

  2. 公司客户端平台（pc端、手机端、ipad端、微信平台、业务系统后台）上捕获的用户操作行为日志数据

    进入到购物平台起所有的点击行为都被以日志的形式存储下来，点击浏览、关键字搜索、加入购物车、查看或评论、确认收货、收藏、提交订单、确认支付、退货…. ） （最常见）

  3. 订单数据

  4. 爬虫爬取的数据

  5. 数据买卖：上海大数据联盟

  6. 电器设备的实验数据：航空发动机设备试验数据

  7. 医药或化工的实验

  8. 银行或支付公司的交易数据

  9. 移动手机设备通信数据

  10. 交通数据

  11. 医疗数据 ：一天新增医疗数据4T

  12. 超市购物数据：上海市各大超市的蔬菜数据

七、数据量分析

  1. 公司客户端平台捕获的用户行为日志数据

  • 每天新增日志：

    ​ .1. 多少条 :3千万-6千万条

    ​ 2.多少G:通常一条用户行为日志的字段数量为10-30个，划算成kb约为 0.5kb-2kb ,假如按照平均每条日志1kb计算，则 3千万-6千万条 为30G ~60G
